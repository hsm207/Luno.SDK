# RFC 003: Domain Exception Hierarchy

**Status:** Draft
**Date:** 2026-03-04

## 1. Overview
This RFC proposes a standardized domain exception hierarchy for the Luno SDK. It establishes `LunoException` as the root class and defines specific derived exception types for various failure domains, such as security, data mapping, and validation. The goal is to provide developers with strongly-typed, predictable error handling without exposing underlying infrastructure details.

## 2. Motivation
Currently, the SDK throws a mix of standard .NET exceptions and raw API exceptions generated by Kiota. This forces developers to catch generic exceptions or sift through HTTP status codes to determine the cause of a failure. A robust SDK must map low-level infrastructure failures into domain-specific, actionable exceptions that align with the Clean Architecture principles of the `Core` layer.

## 3. Future State
Developers will be able to catch specific exceptions corresponding to different failure categories, making error handling more semantic and deterministic.

```csharp
try
{
    var balances = await client.Accounts.GetBalancesAsync();
}
catch (LunoUnauthorizedException ex)
{
    // Handle invalid API keys (401)
}
catch (LunoForbiddenException ex)
{
    // Handle insufficient API key permissions (403)
}
catch (LunoMappingException ex)
{
    // Handle invalid or unexpected API responses
}
catch (LunoException ex)
{
    // Handle any other Luno SDK domain errors
}
```

## 4. Goals & Non-Goals
- **Goals:**
    - Establish `LunoException` as the root exception class.
    - Define `LunoSecurityException` with specialized `LunoUnauthorizedException` (401) and `LunoForbiddenException` (403) classes.
    - Define `LunoDataException` with specialized `LunoMappingException` for data transformation failures.
    - Map HTTP errors to domain exceptions via the `LunoErrorHandlingAdapter`.
    - Provide rich contextual details (e.g., the exact DTO type that failed to map) in the exception metadata.
- **Non-Goals:**
    - Wrapping non-Luno framework exceptions (e.g., `TaskCanceledException`, `HttpRequestException` due to network failure) unless they are semantic domain errors.
    - Overcomplicating the hierarchy with exceptions for every single HTTP status code.

## 5. Proposed Technical Design

### The Hierarchy
All domain exceptions must inherit from `LunoException`, residing in the `Core` layer (`Luno.SDK.Core.Exceptions`).

- `LunoException`
  - `LunoSecurityException`
    - `LunoUnauthorizedException` (Mapped from HTTP 401)
    - `LunoForbiddenException` (Mapped from HTTP 403)
  - `LunoDataException`
    - `LunoMappingException` (Thrown when parsing or mapping DTOs to Domain Entities fails)

### Mapping Logic (LunoErrorHandlingAdapter)
The `LunoErrorHandlingAdapter` (in the `Infrastructure` layer) intercepts raw HTTP responses and translates specific semantic failures into domain exceptions.

| HTTP Status Code | Original Exception | Target Domain Exception |
| :--- | :--- | :--- |
| `401 Unauthorized` | `ApiException` | `LunoUnauthorizedException` |
| `403 Forbidden` | `ApiException` | `LunoForbiddenException` |

Unmapped HTTP errors or internal server errors (`500`) will not be explicitly wrapped unless they are a defined domain behavior; otherwise, the original exception or a generic `LunoException` may be bubbled up based on context.

### Mapping Failure Context
When `LunoMappingException` is thrown, it must include context about what failed.

```csharp
public class LunoMappingException : LunoDataException
{
    public Type? DtoType { get; }

    public LunoMappingException(string message, Type? dtoType = null)
        : base(message)
    {
        DtoType = dtoType;
    }
}
```

## 6. Behavioral Specifications

### Scenario: Security Access Denied
- **Given:** A valid request sent with insufficient API key permissions.
- **When:** The API returns `403 Forbidden`.
- **Then:** `LunoErrorHandlingAdapter` intercepts the failure and throws `LunoForbiddenException`.

### Scenario: Data Parsing Failure
- **Given:** A response from the `/api/1/balance` endpoint missing a critical property (e.g., `account_id`).
- **When:** `AccountMapper` attempts to map the DTO to the `Balance` entity.
- **Then:** The mapper throws `LunoMappingException` specifying the failing DTO type.

## 7. Definition of Done
- **Fully Implemented Exception Classes**: `LunoException` and all derived types are implemented in `Core`.
- **Error Handling Interception**: `LunoErrorHandlingAdapter` correctly maps 401 and 403 status codes.
- **Mapper Hardening**: Mappers fail fast with `LunoMappingException` for missing or malformed required properties.
- **Test Coverage**: All exception classes, mapping scenarios, and adapter interceptions are covered by unit tests (100% compliance).
